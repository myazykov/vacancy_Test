# Реестр требований

## Статусы требований

| Статус | Значение |
|--------|----------|
| **Confirmed** | Прямо следует из вводных задачи, не требует уточнения |
| **Derived** | Выведено аналитиком из контекста, требует валидации |
| **Open** | Требует уточнения (ссылка на Open Question) |

---

## 1. Функциональные требования

### 1.1 Единый реестр договоров

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| FR-001 | Система должна хранить информацию о двух типах договоров: публичная оферта (Offer) и индивидуальный договор (Self-form) в едином реестре | Must | Confirmed | Цель 2: "хранить логи подписанной оферты и сканы подписанных договоров в одном канале" |
| FR-002 | Реестр должен содержать для каждого договора: тип, статус, контрагент, дата, источник, ответственный, документ/ссылка | Must | Derived | Анализ текущей структуры Airtable Agreements + потребности Legal |
| FR-003 | Данные об акцепте оферты должны автоматически передаваться из бэкенда порталов (SellerDrive, Emexdwc) в реестр | Must | Confirmed | "Хранить логи подписанной оферты с сайта... в одном канале" |
| FR-004 | Подписанные Self-form agreements из DocuSign должны попадать в реестр (метод загрузки — развилка, см. TO-BE раздел 4.3) | Must | Confirmed | Цель 2 + текущая проблема покрытия ~40% |
| FR-005 | Каждый договор в реестре должен быть привязан к контрагенту через единый идентификатор (Master Counterparty ID) | Must | Derived | Необходимо для связи Agreements ↔ Payments |

### 1.2 Workflow согласования Self-form Agreement

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| FR-010 | Система должна обеспечить приём запросов от Sales на согласование Self-form agreement (канал — развилка, см. TO-BE раздел 4.2) | Must | Confirmed | "На почту могут юристам приходить запросы на подписание Agreements от менеджера из продаж" |
| FR-011 | Каждый запрос должен автоматически получать дату поступления и дедлайн SLA | Must | Derived | Цель 3: "SLA желаемый — 1 неделя" |
| FR-012 | Система должна фиксировать историю всех действий участников по каждому запросу (Audit Log) | Must | Confirmed | Цель 1: "чтобы хранилась история действий от всех участников" |
| FR-013 | Система должна отправлять уведомления юристу: при поступлении нового запроса, при приближении SLA-дедлайна, при ответе контрагента | Should | Derived | Необходимо для соблюдения SLA |
| FR-014 | Система должна поддерживать эскалацию: если SLA-дедлайн превышен — уведомление руководителю Legal | Should | Derived | Стандартная практика SLA-менеджмента |

### 1.3 Единая точка входа для Legal

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| FR-020 | Юристы должны иметь единый интерфейс (Airtable Interface) для просмотра всех запросов, договоров, статусов и истории | Must | Confirmed | Цель 1: "единую точку входа для работы юристов" |
| FR-021 | Интерфейс должен поддерживать фильтрацию по: статусу, контрагенту, типу, ответственному юристу, SLA-статусу | Should | Derived | Стандартный набор для рабочего интерфейса |
| FR-022 | Интерфейс должен обеспечить доступ к документу (файл или ссылка на DocuSign) прямо из карточки договора | Must | Derived | Исключает переключение между системами |

### 1.4 Связь с оплатами (Pre-payment Check)

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| FR-030 | Перед одобрением оплаты система должна проверять наличие действующего (Active) договора с контрагентом | Must | Confirmed | Цель 3: "перед оплатой система делает проверку по наличию подписанного договора" |
| FR-031 | Результат проверки (Pass / Warning / Block) должен быть виден Approver в карточке платежа | Must | Derived | Необходимо для принятия решения |
| FR-032 | Режим проверки (soft: предупреждение / strict: блокировка) должен быть настраиваемым (развилка, см. TO-BE раздел 4.5) | Should | Derived | Фазированный запуск — начинаем с soft |
| FR-033 | При блокировке оплаты (strict mode) должна быть процедура override с одобрением уполномоченного лица | Should | Derived | Исключительные случаи при неполном реестре |

### 1.5 Прослеживаемость (трассировка)

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| FR-040 | Система должна обеспечить прослеживаемую цепочку: контрагент → договор → инвойс → оплата | Must | Confirmed | "Задача от финансов — сделать процесс с прослеживаемой цепочкой полной" |
| FR-041 | Информация о действующем договоре должна быть доступна из карточки Beneficiary в Airtable Payments (через Synced Table / Lookup) | Must | Derived | Техническая реализация FR-030 |
| FR-042 | Инвойсы из Odoo Documents должны быть привязаны к контрагенту и (при наличии) к договору | Could | Derived | Замыкает цепочку прослеживаемости |

## 2. Нефункциональные требования

| ID | Требование | Приоритет | Статус | Источник |
|----|-----------|-----------|--------|----------|
| NFR-001 | Целевой SLA согласования Self-form agreement: ≤ 7 дней (от Requested до Registered) | Must | Confirmed | "SLA желаемый — 1 неделя" |
| NFR-002 | Решение должно использовать стандартные механизмы Airtable (Linked Records, Synced Tables, Automations, Interfaces) для обеспечения совместимости с обновлениями платформы | Must | Confirmed | Вводная: "сохранить совместимость с дальнейшими обновлениями" |
| NFR-003 | API-интеграции должны быть изолированы в middleware (n8n или отдельный сервис), не встроены напрямую в бэкенд порталов | Should | Derived | Снижение coupling, упрощение поддержки |
| NFR-004 | Время передачи данных об акцепте оферты: не более 5 минут (event-driven) или 60 минут (scheduled sync) | Should | Derived | Баланс между realtime и реализуемостью |
| NFR-005 | Автозагрузка из DocuSign: не более 5 минут от момента completed до появления записи в Airtable | Should | Derived | Текущий показатель: 1-5 дней |
| NFR-006 | Все интеграции должны обрабатывать ошибки: retry с exponential backoff, dead letter queue, алертинг | Must | Derived | Стандарт надёжных интеграций |
| NFR-007 | Система должна поддерживать текущий объём: ~200 поставщиков + ~500 покупателей + рост | Should | Confirmed | Вводная |
| NFR-008 | Конфиденциальность: при использовании внешних LLM API для парсинга email — данные договоров не должны использоваться для обучения модели (zero data retention) | Must | Derived | Договоры содержат коммерческую тайну |

## 3. Ограничения (Constraints)

| ID | Ограничение | Источник |
|----|-------------|----------|
| CON-001 | Порталы SellerDrive и Emexdwc входят в контур компании — доработки зависят от нас | Вводная |
| CON-002 | Airtable (Agreements и Payments) можно дорабатывать — добавлять таблицы, поля, Automations | Вводная |
| CON-003 | Интервью со стейкхолдерами не проводятся — решение строится с вариативностью по ключевым развилкам | Условия тестового задания |
| CON-004 | Решение должно сохранять совместимость с обновлениями Airtable | Вводная |

## 4. Трассируемость требований → GAP'ов → решений

| GAP | Требования | Решение (TO-BE) |
|-----|-----------|-----------------|
| G1: Данные оферты изолированы в БД порталов | FR-001, FR-003 | Развилка 1: Event-driven / Scheduled sync / iPaaS |
| G2: Покрытие реестра ~40% | FR-001, FR-004, FR-005 | Развилка 3: Автозагрузка из DocuSign + ретроспективная загрузка |
| G3: Нет связи договор ↔ оплата | FR-030, FR-031, FR-040, FR-041 | Развилка 4: Synced Table + Lookup + Развилка 5: Soft/Strict |
| G4: Email-эскалация без трекинга | FR-010, FR-011, FR-012 | Развилка 2: Dedicated mailbox + AI / Form / Гибрид |
| G5: Нет единой точки входа для Legal | FR-020, FR-021, FR-022 | Airtable Interface (единый для всех потоков) |
| G6: Разрозненные архивы | FR-040, FR-042 | Phase 3: Odoo ↔ Airtable sync |
| G7: Неполнота данных Beneficiary | FR-041 | Synced Table + Lookup, validation rules (Phase 3) |
