1. Контекст и допущения

1.1 Стейкхолдеры

⚖️ Legal (Заказчик)

Согласование Self-form agreements. Боль: 6 систем, нет единой точки входа, нет истории действий.

💰 Finance (Со-заказчик)

Оплаты через Airtable Payments. Нужна цепочка «договор→инвойс→оплата», проверка перед платежом.

🛒 Sales

Инициируют Self-form через email. Нет видимости статуса. Готовность к изменениям — открытый вопрос.

🏢 Контрагенты

~200 поставщиков (SellerDrive) + ~500 покупателей (Emexdwc).

1.3 Допущения

Д1: Порталы — веб-приложения с бэкендом и БД. Акцепт фиксируется структурированно. Оба внутри контура.
Д2: Airtable можно дорабатывать. Сохранять совместимость с обновлениями: стандартные механизмы.
Д3: Интервью не проводятся. Решение с вариативностью + открытые вопросы.
Д4: Готовность Sales/Legal/Finance к изменениям — неизвестна. Владелец Payments — неизвестен.

2. AS-IS

2.1 Акцепт оферты

🌐 Контрагент → Портал
  1. A1 Авторизуется на портале
  2. A2 Акцептует оферту (фронтенд)
  3. A3 Бэкенд фиксирует в БД (timestamp, user ID, версия)
  4. ⚠️ Конец. Данные в БД. Нет интеграции. Legal/Finance не видят.

2.2 Self-form Agreement

📨 Sales → Legal → DocuSign → Airtable
  1. B1 Контрагент предлагает свой формат
  2. B2 Sales → email юристам (свободная форма)
  3. B3 Юрист получает в почте. Нет SLA.
  4. B4 Итерации email/ad-hoc
  5. B5 Подписание в DocuSign (вручную)
  6. B6 Юрист вручную: скачивает PDF → запись в Airtable
  7. ⚠️ ~60% не попадают в реестр. SLA: 2 нед.

2.3 Оплаты

💰 Odoo + Airtable Payments
  1. C1 Инвойсы → Odoo Documents
  2. C2 Оплата → approval flow
  3. C3 Beneficiary: реквизиты, нет ссылки на договор
  4. ⚠️ Нет проверки договора перед оплатой.

3. GAP-анализ

G1 · CriticalДанные акцепта изолированы в БД порталов

Факт акцепта не передаётся во внутренние системы. Legal не может подтвердить договорные отношения.

G2 · CriticalПокрытие реестра ~40%

Загрузка ручная. ~60% пропускается. Нет напоминаний.

G3 · CriticalНет связи «договор — оплата»

Payments и Agreements изолированы. Beneficiary без ссылки на договор.

G4 · HighEmail — неуправляемый канал

Нет трекера, SLA, видимости для Sales.

G5 · HighНет единой точки входа для Legal

Email + DocuSign + Airtable. Нет единого workspace.

G6 · HighРазрозненные архивы

Договоры/инвойсы/оплаты — три системы без связей.

G7 · MediumНеполнота данных Beneficiary

Пустые поля, нет привязки к договору.

4. TO-BE (с развилками)

Подход: по каждой точке — варианты с «за/против». Выбор после валидации.

4.0 Стратегическая: Airtable vs Odoo Contracts

Выбор платформы contract management

Odoo Contracts даёт нативную связку в одной экосистеме. Airtable — быстрый старт.

Стратегия Aдля анализа

Расширить Airtable + интегрировать

✅ За
  • Быстрый старт
  • Гибкость no-code
  • Мин. change management
❌ Против
  • Интеграции через middleware
  • Нет нативного workflow
  • Лимиты при масштабировании
Стратегия B

Миграция в Odoo Contracts

✅ За
  • Нативная цепочка
  • Workflow, шаблоны
  • Единая БД контрагентов
❌ Против
  • Миграция + переобучение
  • Дольше и дороже

💡 Рассмотреть как Phase 4.

6. Roadmap

Phase 1 · 4–6 нед.Фундамент
  • Расширение Airtable: Offer Logs, Self-form Tracker, Audit Log
  • Legal Interface
  • Dedicated mailbox + базовый парсинг
  • SLA-автоматизации
  • Ретроспективная загрузка из DocuSign
  • Data cleansing, Master ID

→ Покрытие 70–80%. SLA измеряется.

Phase 2 · 4–6 нед.Интеграции
  • n8n: развёртывание
  • Порталы → Airtable (событие акцепта)
  • DocuSign → Airtable (Pending Review)
  • AI-парсинг email
  • Synced Table → Payments
  • Pre-payment check: soft

→ Покрытие 95%+. Автозагрузка.

Phase 3 · 3–4 нед.Прослеживаемость
  • Odoo↔Airtable: инвойсы + бэкап PDF
  • Strict mode
  • Dashboard Finance
  • Reporting, validation

→ 100%. ≤7 дн. Compliance.
